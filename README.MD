#Z8 Task Tracker



##Тестовое приложение на базе Z8 Template.



##Краткое описание:



Удалось реализовать все задачи. Backend и Frontend в полном объеме. Через Admin доступны компоненты task tracker (backend часть) и документы (frontend часть).



##Возможные отличия от поставленной задачи:



###Представление "Проекты"



В ТЗ явно не указано наличие отдельного представления "Проекты", однако было принято решение добавить его как самостоятельную сущность.

Проекты видны всем пользователям системы, однако:

создавать проекты может только пользователь с ролью Admin;

добавлять пользователей в проекты может только пользователь с ролью Admin.

При необходимости компонент "Проекты" можно скрыть для обычных пользователей для более строгого соответствия ТЗ.



###Логика работы с задачами



###ID задачи



ID задачи генерируется автоматически при создании и является уникальным.

В связи с этим поле ID было заблокировано для редактирования пользователем.



###Создатель задачи



При создании задачи автоматически назначается её создатель - пользователь, создавший задачу.

Поле "Создатель" также было заблокировано для редактирования, так как логически создатель задачи изменяться не должен.



###Код проекта



Проект определяется уникальным кодом.

Код проекта можно изменять в представлении "Проекты".

В представлении "Все задачи" код проекта изменить нельзя, так как он связан с выбранным проектом. При смене проекта у задачи код проекта автоматически обновляется в соответствии с выбранным проектом.



###Сущность "Пользователь"



В ТЗ не до конца очевидно, требуется ли переопределение внутренней логики паролей Z8.

Было принято решение не вмешиваться глубоко во внутреннюю систему авторизации платформы.

В представлении "Пользователи" используются поля:



\*Логин

\*Имя

\*Фамилия



При создании нового пользователя вход в систему выполняется:



\*с указанным логином;

\*с пустым паролем.



После первого входа система предлагает изменить пароль стандартными средствами Z8.



##Сторонние библиотеки:



Для подсветки XML используется CodeMirror 5.



Файлы библиотеки добавлены в проект локально:



\* `src/main/js/js/lib/codemirror/codemirror.js`

\* `src/main/js/js/lib/codemirror/mode/xml/xml.js`

\* `src/main/css/codemirror.css`



Файлы подключены в сборку через `.buildorder`.



##Изменения стандартной сборки:



В стандартную сборку был добавлен workaround для CSS-сборки на Windows.

Причина: стандартная задача `minifyCss` в текущем окружении формировала некорректный путь к CSS-файлу и падала при сборке.

Для обхода проблемы задача `minifyCss` отключена, а CSS после `concatCss` копируется отдельной задачей.

Если используется другая среда, где стандартная CSS-сборка работает корректно, этот workaround можно удалить.



##Известные особенности/проблемы:



###Создание и удаление пользователей



При создании пользователя через `trackerUser` ему автоматически назначаются роль `User` и пункты меню.



Если сразу после создания перейти в системный экран `Users` и попытаться удалить пользователя, в некоторых случаях может возникнуть ошибка внешнего ключа. При повторной попытке через небольшой промежуток времени пользователь удаляется корректно.

Чтобы избежать возможных ошибок с удалением пользователей, была реализована логика: при удалении пользователя из проекта автоматически обновляются связанные задачи и история изменений. Удалить пользователя до его удаления из проектов - нельзя.



###Statistics является вычисляемой моделью с aggregation/groupBy

и не создаётся как физическая таблица в БД.



###По задаче с добавлением аудиоплеера при загрузке аудиофайла:

Обновление плеера выполняется через refresh в соответствующем поле после загрузки файла.
Сделать так, чтобы при загрузке файла плеер появлялся без каких-либо обновлений не удалось.



###История изменений задачи task tracker:

Для апдейта истории необходимо обновить страницу или конкретно элемент ExHistory на странице с задачей.
